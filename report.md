### 数据结构

```c
// 结构体
struct FRAME { //定义数据链路层传输的帧的结构，用于在发送方和接收方之间传递数据和控制信息。
    unsigned char kind;  // 标识帧的类型。宏定义隐含为 FRAME_DATA(1 数据帧), FRAME_ACK(2 确认帧), FRAME_NAK(3 否定应答帧)。
    unsigned char ack;   // 确认号。对于数据帧，它是捎带确认机制中期望收到的上一个成功帧的序号；对于ACK/NAK帧，它是上一个成功接收的序号。
    unsigned char seq;   // 序列号。当前发送的数据帧的序号，用于滑动窗口的顺序控制。
    unsigned char data[PKT_LEN]; // 数据载荷。存放从网络层获取的实际数据包，大小为 PKT_LEN (256字节)。
    unsigned int  padding;       // 填充字段。确保整个结构体大小满足特定要求（在计算CRC前补齐字节）。
};

// 全局变量
// 状态标志变量 
static int no_nak = 1;  // NAK发送控制标志。1表示当前可以发送NAK，0表示已发送过NAK，防止针对同一乱序错乱连续发送多个NAK。
static int phl_ready = 0; // 物理层就绪标志。1表示物理层空闲可以发送数据，0表示忙碌。
// 窗口控制变量
static int max_seq_num = 7; // 发送窗口的最大尺寸（序列号范围大小）。默认7，可通过命令行参数修改。
static unsigned char send_window_l = 0; // 发送窗口左沿（下界）。代表最早发送但尚未收到ACK确认的帧序号。
static unsigned char send_window_r = 0; // 发送窗口右沿（上界）。代表下一个即将发送的新数据帧的序号。
static unsigned char frame_expected = 0; // 接收窗口。由于是GBN协议，接收窗口大小为1，代表接收方当前期望接收的下一个按序到达的帧序号。
// 缓冲池变量 
static unsigned char buffer[ABSOLUTE_MAX_SEQ_NUM+1][PKT_LEN]; // 发送方的缓存区。用于暂存已发送但未确认的数据包，以便超时或收到NAK时重传。
static unsigned char nbuffered = 0; // 当前缓存区中已缓冲的数据包数量（即当前发送窗口内正在等待确认的帧的数量）。
// 主函数中的变量
int event; //事件类型变量。用于存储从 wait_for_event() 获取到的系统事件（如网络层就绪、收到帧、超时等）。
int arg; // 事件参数变量。与 event 配合使用，例如当发生 DATA_TIMEOUT 时，arg 存储发生超时的那一帧的序号。
struct FRAME f; //帧结构体变量。作为接收缓冲，用于存放从物理层(recv_frame)接收到的最新一帧的数据。
```

### 算法原型

1. 初始化：解析命令行参数设定最大窗口大小（默认为7），初始化协议层，关闭网络层接收（直到物理层准备好）。
2. 事件等待：进入无限循环 for(;;)，调用 wait_for_event() 阻塞等待系统事件。
3. 事件分发与处理（switch(event)）：
    1.网络层就绪 (NETWORK_LAYER_READY)：获取新数据包放入缓存，发送对应的数据帧，窗口右沿递增，停止ACK计时器（因为确认已捎带）。
    2.物理层就绪 (PHYSICAL_LAYER_READY)：标记 phl_ready = 1。
    3.收到帧 (FRAME_RECEIVED)：
        - CRC校验失败：丢弃，不回应。
        - 收到NAK帧：立即从NAK指示的序号开始，重传发送窗口内的所有帧。
        - 收到数据帧：
            - 如果是按序到达的期望帧：将其交给网络层，期望序号+1，启动ACK计时器（等待捎带），重置NAK发送标志。
            - 如果是乱序帧：且当前允许发送NAK，则发送NAK帧请求重传。
    4. 处理确认信息：无论收到何种帧，只要其携带的 ack 落在发送窗口内，则将发送窗口左沿右移，释放缓存，并停止对应帧的计时器。
4. ACK超时 (ACK_TIMEOUT)：说明一直没有己方数据帧可以捎带确认，被迫发送一个纯ACK帧。
5. 数据超时 (DATA_TIMEOUT)：典型的GBN操作。从发送窗口的左沿（未确认的第一个帧）开始，把窗口内所有缓存的数据帧重新发送一遍。
6. 流量控制：判断当前缓冲区包数 nbuffered < max_seq_num 且物理层就绪时，开启网络层（允许其下发新数据）；否则关闭网络层，实现背压限流。循环回到步骤2。

## 实验结果分析

### 无差错传输功能的实现

本实验通过追加 4 字节的 CRC-32 校验码来实现信道的检错功能。具体来讲：
在发送端，程序利用实验库提供的 `crc32(p, len)` 函数，对包含帧头和有效载荷的完整数据计算出 32 位校验和，并附加在帧尾发送。
在接收端，接收方将接收到的完整数据帧（包括帧头、有效载荷和附加的校验和）整体通过函数 `crc32()` 进行“模2除法”。若返回结果为 0，说明传输过程无差错，正常接收并提取数据；若返回结果非 0，说明帧在物理信道中发生了误码。此时接收方直接丢弃该数据帧，不发送相应的 ACK 帧。发送方在重传定时器超时后，会自动重新发送该数据帧，直到接收方接收到正确的数据帧并返回 ACK 为止。

CRC-32 的 33 位生成多项式（与 IEEE 802.3 以太网标准相同）**足以将所有单比特错误、双比特错误、奇数个错误以及长度小于等于 32 位的突发错误检测出来**，从而在误码率为 $10^{-5}$ 的不可靠物理信道上，向网络层提供了可靠的无差错传输服务。

### 健壮性

1. 用户交互方面，通过拦截非法参数（大于255和小于等于0），确保了手动设置窗口情况下程序的健壮性。
2. 数据正确性方面，通过严格的CRC-32校验码确保接收数据的正确性。
3. 时序状态方面，通过`frame_expected`变量确保交给网络层的数据没有断层。
4. 死锁状态方面，通过ACK_TIMER和DATA_TIMER确保系统在没有接收到ACK帧的情况下能正确冲传播、恢复工作。


### 协议参数的选取

已知物理层仿真程序提供的通信信道具有如下特性：带宽 $C = 8000 \text{ bps}$，单向传播时延 $T_{prop} = 270 \text{ ms}$，误码率为 $10^{-5}$。
网络层传递给数据链路层的分组（Payload）长度固定为 256 字节。封装成帧时，需附加 3 字节帧头与 4 字节 CRC 校验和。
由此可得：单一数据帧总大小 $L_{data} = 3 + 256 + 4 = 263 \text{ Bytes} = 2104 \text{ bits}$。纯 ACK 帧大小设定为 2 字节 = 16 bit。
根据公式 $T = L / C$ 可得：数据帧发送时延 $T_{data} = 2104 / 8000 = 263 \text{ ms}$，纯 ACK 帧发送时延 $T_{ACK} = 16 / 8000 = 2 \text{ ms}$。

我们基于以上物理参数进行严格推导，并结合软件仿真环境的实际情况选取协议参数：

#### 滑动窗口的大小 (Window Size)

为了最大化信道利用率，实现“跑满链路”（即信道不空闲），发送方的滑动窗口大小 $W$ 必须能够覆盖一个完整的往返时间（RTT）内可以发送的帧数，即：
$$W \ge \frac{T_{data} + 2 \times T_{prop} + T_{ACK}}{T_{data}}$$
代入数据计算可得：
$$W \ge \frac{263\text{ms} + 2 \times 270\text{ms} + 2\text{ms}}{263\text{ms}} = \frac{805\text{ms}}{263\text{ms}} \approx 3.06$$
**分析**：尽管 3.06 非常接近 3，但如果设定序列号空间为 2 bit（GBN 协议最大窗口 $W=2^2-1=3$），发送完 3 个帧仅需 $3 \times 263 = 789 \text{ ms}$，此时第一个 ACK 还需等待 16 ms 才能到达，这会导致信道出现微小的空闲，无法达到理论 100% 利用率。因此，为了完全发挥信道性能，必须向上取整使 $W \ge 4$。
由此，序列号空间至少需要 3 bit。本协议遵循惯例设置默认滑动窗口大小为 $W=7$（即 3 位序号的最大值）。

#### 重传定时器的时限 (DATA_TIMER)

在本协议的软件实现中，由于 `start_timer()` 是在开始发送帧数据时同步调用的，因此定时器的理论下限应当覆盖从“开始发送帧的第一比特”到“接收完 ACK 帧的最后一比特”所需的完整时间，外加接收方的捎带等待时间（`ACK_TIMER`）：
$$T_{理论下限} = T_{data} + 2 \times T_{prop} + T_{ACK} + T_{ACK\_TIMER}$$
假设接收方最大等待 $150\text{ms}$，理论下限为 $263 + 540 + 2 + 150 = 955 \text{ ms}$。
**安全冗余的考量**：理论上设置 1000ms 即可，但在实际测试中发现，仿真程序运行于 Windows 系统之下，TCP Socket 通信及多线程进程切换存在明显的调度抖动。正如实验指导书（表2）所警示的情况（`sleep 15 ms, but be awakened 61 ms later`），操作系统级别的调度误差往往高达50ms。若冗余设置过小，极易引发伪超时，导致 GBN 协议发生灾难性的回退重传。
**因此，经过实际上机调试与性能对比，我们将重传定时器时限最终设定为 1200ms**，在保证及时发现丢包的同时，有效吸收了操作系统的调度抖动。

#### ACK搭载定时器的时限 (ACK_TIMER)

ACK 搭载定时器用于在接收到数据后，若网络层暂时无法产生新分组进行“捎带确认”，则强制触发纯 ACK 帧的发送，以防发送方超时。但是，若时限过短，网络层几乎没有机会产生新分组，协议将频繁发送纯 ACK，浪费信若时限过长，会导致 RTT 严重拉长，极易触发发送方的重传定时器），引发不必要的重传。
综上考量，参考Windows操作系统微软官方文档（TCP/IP Registry Values）中TcpDelAckTicks 注册表项默认配置的延迟时间，我们将设为 **200ms**。这个数值既留出了足够的时间博取捎带概率，又将控制权妥当地把控在重传时限的安全范围内。

### 理论分析

#### 无差错环境下的最大信道利用率

$BER=0$时，由于协议头部固定开销（7字节/帧）的存在，理论最大信道利用率$$U_{max,BER=0}=\frac{有效载荷长度}{数据帧总长度}=\frac{256}{263}\approx 97.34%$$。

#### 有误码条件下理想情境中的最大信道利用率

$BER=p$时，单帧错误率是$P_f=1-(1-p)^L$，其中L是数据帧的长度（单位为比特），取2104。
在GBN协议中，如果发生1个帧错误，发送方需要重传出错的帧以及后面的$W-1$个帧。所以，出错1次的代价是耗费$W$个包的时间。理论最大信道利用率为$$U_{max,BER\neq 0}= U_{max,BER=0} \times \frac{1-P_f}{1+(W-1)P_f}$$。
当$p$取$10^{-4}$,$W$取默认值7时，$P_f\approx 0.1898$，$U_{max,BER\neq 0}=0.9734 \times \frac{1-0.1898}{1+6\times 0.1898} \approx 36.87%$；
当$p$取$10^{-5}$,$W$取默认值7时，$P_f\approx 0.0208$，$U_{max,BER\neq 0}=0.9734 \times \frac{1-0.0208}{1+6\times 0.0208} \approx 84.74%$；

### 存在的问题

上述性能测试均能正常运行，但是项目2和项目4的信道利用率均与理论值84.74%差距较大。将窗口大小$W$修改为10后再次执行项目4的命令，两个窗口的信道利用率分别达到71.14%和71.58%，因此初步判断问题可能在于信道闲置；可惜当窗口大小修改至15后，两个窗口的信道利用率分别为72.29%和71.44%，或许由于窗口增大导致重传成本也相应增大，信道利用率没有进一步提升。
联想到最初通过实验手册给出的传播时延和传输时延计算出的W大小仅为3.06，但实际设置W=7远远没有跑满信道这一事实。推测由于定时器为捎带传送和网络抖动等因素设置了大量冗余，变相增长了时延，所以当W增大至10左右时，信道利用率才达到相对较高的水平。于是将DATA_TIMER单独减小至1000后，发现两个窗口的信道利用率分别为66.29%和60.65%，略有提升；将ACK_TIMER单独减小至100后，信道利用率为53.47%和52.39%，没有提升。叠加W=10，DATA_TIMER=1000,ACK_TIMER=100后，信道利用率仍然停留在64.74%和62.14%。这进一步证明想要信道利用率逼近理论值，需要这三项参数达到合理的平衡，不能通过单纯的增大W，减少定时器时长来达到提升信道利用率的效果。
因此猜测正常误码率与洪水发参前提下，信道利用率远低于理论值是窗口和定时器时长设置不合适的结果。但由于时间原因，很遗憾没能继续展开探索最佳参数设置。

## 研究和探索的问题

1. CRC校验能力
    (1)数学上确实没有100% 的绝对安全，但CRC-32能够100% 检测出所有单比特错误、双比特错误、奇数个比特错误，以及长度小于等于 32 位的突发错误。除了以上三种必定能检测出的情况，只有当错误极其巧合地满足特定多项式倍数时才会漏检，而这种巧合发生的概率是极低的，具体推导见下:
    (2)假设物理链路速率为 $R = 8000 bps$，平均帧长为 $L = 2104bits$。每天使用率50%，则每天发送的帧数为：$N = (8000bps / 2104bits) \times (24 \times 3600 \times 50%) \approx 1.64\times 10^6$ 帧/天。假设在极端恶劣环境下（误码率极高），所有帧都被破坏成了完全随机的噪声。此时，随机数据恰好能算出正确 CRC-32 校验和的概率是 $P = 2^{-32} \approx 2.33 \times 10^{-10}$。每天发生漏检的概率（期望值）为 $E = N \times P \approx 1.64 \times 10^6 \times 2.33\times 10^{-10} \approx 3.82\times 10^{-4}$ 次/天。
    即使在最极端的“全噪声”假设下，大约需要2617天才会漏检一次。如果链路传输正确率较高（误码率在 $10^{-9}$ 级别），漏检发生的时间将拉长到几千年甚至上万年。并且，上述计算假设错误是完全随机的。但真实网络中的错误通常是“突发性”的（比如电磁干扰瞬间破坏几十个 bit），而CRC对这种突发错误的检测能力极强，漏检率远低于$2^{-32}$。若还要进一步降低错误概率，还可以在网络层/应用层增加更强校验（如TCP伪首部校验和，或应用层使用 MD5/SHA-1 哈希校验），但这极大地增加了 CPU 计算开销（MD5 耗时远超 CRC），且增加了首部字节开销（降低了有效吞吐量）。
    然而，CRC 只能防“无意”的物理噪声，防不了恶意篡改（比如，黑客可以轻易伪造正确的 CRC）。此外，网卡硬件 Bug 或内存翻转也可能在软件校验之后破坏数据。
2. CRC校验和的计算方法
    (1)查表法与手工模2除法完全等效，因为模2除法（本质是异或 XOR 操作）满足线性组合律。
    查表法比手工模2除法快得多的原理如下：手工计算是按位处理的，每一次移位都需要判断和异或，如果模拟手工计算实现模2触发，CPU分支预测开销极大；而软件实现采用的查表法是按字节处理的。算法提前把这一个字节对应的256种情况对32 位寄存器造成的“异或影响”计算出来存入数组，每次直接吃进 8 个 bit，速度提升了近 8 倍。
    (2)构造 `crc_table[256]` 的 C 语言程序核心代码如下：
```c
    unsigned int crc_table[256];
    void make_crc_table() {
        unsigned int c;
        for (int n = 0; n < 256; n++) {
            c = (unsigned int)n;
            for (int k = 0; k < 8; k++) {
                // 如果最低位是 1，则右移并与生成多项式异或（这里多项式被反转为 0xEDB88320）
                if (c & 1) {
                    c = 0xEDB88320 ^ (c >> 1);
                } else {
                    c = c >> 1;
                }
            }
            crc_table[n] = c;
        }
    }
```
    (3)x86 上 CRC32 比 CRC16 不会多花一倍 CPU 时间，甚至可以说时间几乎一模一样。这是因为现代 x86 处理器是 32 位或 64 位架构。对 ALU来说，执行一次 16 位异或指令和执行一次 32 位异或指令，都需要且仅需要 1 个时钟周期。算法的性能瓶颈在于内存读取而不是 ALU 计算宽度。
    (4)RFC1662 的 `pppfcs32(fcs, cp, len)` 需要三个参数的原因是实验提供的 `crc32(buf, len)` 只能一次性对连续内存块进行计算。RFC 的函数多了一个 `fcs` 参数（代表上一步计算的中间校验值）。这是一种流式计算设计！在真实操作系统内核（如 Linux 的 `sk_buff`）中，一个数据包可能被拆分在内存的不同物理页中（非连续）。传入 `fcs` 可以让系统先算一段，把结果传入下一次调用再算下一段，极其灵活。
2. 程序设计方面的问题
    (1)协议软件跟踪功能的意义：网络协议是高度异步且并发的。如果用GDB 设断点调试，程序一停，定时器还在走，会导致大范围超时，彻底破坏协议运行的时序。所以必须依赖打日志来进行事后重放分析。
    (2)由于标准 C (`time.h`) 的精度只能到秒，自己写get_ms()需要调用系统 API。
    (3)`printf` 风格函数（可变参数 `lprintf`）利用了 C 语言的 `<stdarg.h>` 头文件和堆栈特性来实现：底层通过 `va_list`, `va_start()`, `va_arg()`, `va_end()` 宏，依靠栈指针的偏移，根据第一个参数（格式化字符串中的 `%d`, `%s` 的个数）去依次抓取后面压入栈中的未知数量的参数。`lprintf` 内部实际上是包装了 `vsnprintf`，同时把格式化好的字符串向屏幕（stdout）和日志文件（File 指针）各写一份。
    (4)`start_timer()` 和 `start_ack_timer()` 残留时间处理不同的原因如下：
    `start_timer(DATA_TIMER)`：如果定时器没到期时再次调用它，通常意味着你刚刚重新发送了该帧。此时，定时器必须被重置（清零重新倒计时。因为既然重新发了，就要再给它完整的RTT 时间去等待ACK。
    `start_ack_timer()`：如果在定时器没到期时又收到了一个新帧，再次调用它，此时必须保留残余时间，否则导致发送方误以为上一帧丢了而触发多余的重传。
3. 软件测试方面的问题
   (1)七种测试方案的目的与瞄准的环节：
        - 无误码测试：测试基础状态机的正确性。若失败，说明代码逻辑有基础Bug。
        - 洪水发参测试：测试协议的最大利用率。
        - 有/高误码测试：测试重传机制（DATA_TIMER等）。若失败，说明出错处理逻辑混乱或触发了死锁。
   (2)未覆盖的问题（漏洞盲区）：实验环境下测试误码可能是均匀随机的。真实网络中，错误常是连续发生的（比如雷电等干扰导致连续10个帧报废）。此时NAK机制会崩溃，只能依赖超时。
   (3)更高效的测试方案：通过自动化模糊测试（Fuzzing）结合断言（Assertion），即编写脚本自动随机生成W、误码率参数，彻夜运行（压力测试），并在代码中埋入断言（如 assert(frame_expected <= next_frame_to_send)），一旦状态机非法直接报错，而不是只看利用率。
   (4)程序库的不足与改进建议（对系统架构的思考）：
        不足：当前的测试误码是黑盒随机故障，无法复现特定的错误，比如“精准丢弃某个ACK 帧”来测试特定逻辑。
        建议：在底层库增加故障注入模块，允许测试者编写脚本精准控制哪一帧丢弃、哪一帧超时，这将方便协议开发中的单元测试。
4. 对等协议实体之间的流量控制
   本程序所设计的滑动窗口协议软件已经解决了对等实体之间的流量控制问题。
   具体原理是：发送方维持一个发送窗口W。这意味着发送方在没有收到接收ACK的情况下，最多只能发送W个未确认帧；如果接收方处理太慢，或者由于某种原因无法发送 ACK（虽然本实验中接收端处理很快，但逻辑上是成立的），发送方发出W个包后就会填满窗口。此时，发送方代码会自动停止调用 get_packet()（通过停用网络层），强制发送方停下来，即接收方通过“是否发送 ACK 以及发送多快的 ACK”在物理上钳制了发送方的发送速率，防止接收方被数据洪流淹没。
   此外，本程序还通过 enable/disable_network_layer 实现了跨层流控，防止了网络层产生数据的速度超过链路层的发送能力，避免了发送缓冲区溢出。

5. 与标准协议的对比
    1. 5000公里卫星信道的问题与解决：
    问题：同步卫星单程距离极大，会导致RTT都会急剧增加（几百毫秒甚至更高）。根据我们在10.4中分析的 BDP 原理，此时现有的W=7的小窗口将导致信道利用率趋近于0，发送方绝大部分时间都在等 ACK。此外，卫星信道误码率较高，GBN 的连坐重传会引发灾难性拥塞。
    解决方案：大幅扩展窗口大小（扩大序列号位数），并且抛弃GBN协议，改用SR协议，只重发那一个出错的帧。
    2. 实验协议与 LAPB 标准协议的差距：遗漏了“链路连接管理（Link Setup and Teardown）”。本程序运行开始后双方直接发送数据。但在真实的LAPB中，通信必须经过三个阶段：
        1.建立连接：发送方必须先发 SABM (Set Asynchronous Balanced Mode) 帧，接收方回复 UA (Unnumbered Acknowledgment) 帧，此时链路才算打通，序列号清零初始化。
        2.数据传输：本实验实现的部分。
        3.释放连接：发 DISC 帧断开连接。
    本实验协议只实现了LAPB中的数据传输子集，遗漏了标准协议中不可或缺的状态同步、链路保活监控以及连接生命周期管理功能。

8小时30分钟左右。相比预设的4\~6小时时间开销，多余的调试时间主要来源于对默认误码率和洪水发参情况下，信道利用率远低于理论值的异常情况的调试和探究。

对实验用IDE vscode较为熟悉，实验过程中没有感受到明显困难。

### 编程语言方面的问题

实验开始前认真复习了数据链路层相关协议，实验过程中没有感受到明显困难。

### 开发库方面的问题

在物理层与仿真环境配置阶段，运行实验指导提供的示例程序（如 `stopwait.c`、`gobackn.exe`、`selective.exe`）时，系统抛出了“找不到 MSVCP120.dll”的运行环境报错。经查阅资料确认，这是由于本机较新的 MSVC 2022 环境缺失了旧版的 C++ 运行库。通过额外下载并安装 Microsoft Visual C++ 2013 (x86) 运行库补丁后，成功解决了动态链接库缺失的问题，底层 TCP 通信与时钟中断机制得以正常工作。

1. 通过动手实践，熟悉了GBN协议的原理和应用。
2. 通过对默认误码率和洪水发参情况下，信道利用率远低于理论值的异常情况的探究，认识到网络性能来源于协议开销和硬件带宽的共同作用。
