## 软件设计

### 数据结构

```c
// 结构体（与GBN版本完全一致）
struct FRAME {
    unsigned char kind;  // FRAME_DATA(1), FRAME_ACK(2), FRAME_NAK(3)
    unsigned char ack;   // [SR] 独立ACK：确认号表示具体收到哪一帧（GBN为累积ACK）
    unsigned char seq;   // 当前发送帧的序号
    unsigned char data[PKT_LEN]; // 有效载荷256字节
    unsigned int  padding;
};

// ========== SR新增变量（GBN中没有） ==========
// 发送方新增：
static int acked[ABSOLUTE_MAX_SEQ_NUM+1]; // 标记发送窗口中哪些帧已被独立确认

// 接收方新增（SR核心差异）：
static unsigned char recv_window_r;        // 接收窗口右沿（GBN的接收窗口固定为1）
static int arrived[ABSOLUTE_MAX_SEQ_NUM+1];// 标记接收窗口中哪些帧已到达
static unsigned char recv_buffer[ABSOLUTE_MAX_SEQ_NUM+1][PKT_LEN]; // 乱序帧缓存

// ========== SR窗口约束 ==========
int sr_window_size;  // SR窗口 = (max_seq_num+1)/2，默认4（GBN默认为7）
```

### GBN与SR协议核心差异对照表

| 特性 | GBN（同伴实现） | SR（本实现） |
|------|---------------|------------|
| **接收窗口大小** | 1（仅`frame_expected`） | W = (MAX_SEQ+1)/2（默认4） |
| **乱序帧处理** | 丢弃 + 发NAK | 缓存 + 发独立ACK |
| **ACK语义** | 累积ACK（ACK=n表示0~n全部收到） | 独立ACK（ACK=n表示仅收到帧n） |
| **DATA_TIMEOUT重传** | 重传窗口内全部帧（连坐） | 仅重传超时的单个帧 |
| **NAK重传范围** | 从missing开始重传所有后续帧 | 仅重传missing那一帧 |
| **发送窗口滑动** | ACK到达即连续滑动 | 仅滑动连续已ACK的帧 |
| **窗口大小上限** | W ≤ MAX_SEQ（默认7） | W ≤ (MAX_SEQ+1)/2（默认4） |
| **相同部分** | CRC校验、帧格式、捎带确认、ACK/NAK定时器、流量控制背压 | ←全部相同 |

### 算法原型（SR区别于GBN的关键流程）

1. 初始化：解析命令行参数设定max_seq_num（默认为7），计算SR窗口大小 `sr_window_size = (max_seq_num+1)/2`（默认为4），初始化acked[]和arrived[]数组，初始化协议层，关闭网络层接收。

2. 事件等待：进入无限循环 `for(;;)`，调用 `wait_for_event()` 阻塞等待系统事件。

3. 事件分发与处理（`switch(event)`）：
    1. 网络层就绪 (NETWORK_LAYER_READY)：**与GBN相同**。获取新数据包放入缓存，发送数据帧，窗口右沿递增，停止ACK计时器。
    2. 物理层就绪 (PHYSICAL_LAYER_READY)：**与GBN相同**。标记 `phl_ready = 1`。
    3. 收到帧 (FRAME_RECEIVED) —— **SR核心差异区域**：
        - CRC校验失败：丢弃，不回应（**与GBN相同**）。
        - 收到ACK帧：**【SR差异①】** 标记 `acked[f.ack] = 1`（独立确认），而非GBN的累积确认。
        - 收到NAK帧：**【SR差异②】** 计算 `missing = f.ack + 1`，**仅重传missing这一帧**，而非GBN的从missing开始全部重传。
        - 收到数据帧：**【SR差异③】**
            - 若帧序号落在接收窗口 `[recv_window_l, recv_window_r)` 内 → 缓存至 `recv_buffer[seq]`，标记 `arrived[seq]=1`。然后从 `recv_window_l` 开始连续交付所有已到达的帧给网络层，接收窗口同步右移。启动ACK计时器等待捎带。
            - 若帧序号不在接收窗口内 → 且允许发送NAK，则发送NAK请求缺失帧。
        - 发送窗口滑动：**【SR差异④】** 从 `send_window_l` 开始，连续滑动所有已被ACK的帧（`acked[send_window_l]==1`），逐个停止计时器。而非GBN的一次性累积滑动。
    4. ACK超时 (ACK_TIMEOUT)：**与GBN相同**。发送纯ACK帧。
    5. 数据超时 (DATA_TIMEOUT)：**【SR差异⑤】仅重传超时的单个帧`arg`**，而非GBN的从`send_window_l`开始重传窗口内全部帧。
    6. 流量控制：**与GBN结构相同**，但使用 `nbuffered < sr_window_size`（而非GBN的 `nbuffered < max_seq_num`）控制发送速率。循环回到步骤2。

### SR协议正确性保证

1. **序列号歧义避免**：SR要求窗口大小 ≤ 序列号空间的一半。本实现自动计算 `sr_window_size = (max_seq_num+1)/2`（默认4，序列号空间0~7共8个），确保接收方不会混淆新帧和重传的旧帧。

2. **乱序帧缓存**：`recv_buffer[][]` 按序号索引存储，`arrived[]` 标记已到达状态。每当 `recv_window_l` 对应的帧可用时，连续交付所有连续的已到达帧。

3. **独立重传**：每个帧拥有独立的定时器（由库函数 `start_timer(nr, ms)` 支持），超时时仅重传对应帧，避免了GBN的"连坐"开销。

## 实验结果分析

### 无差错传输功能的实现

本实验与GBN版本共用相同的CRC-32校验实现。在发送端，利用 `crc32(p, len)` 计算32位校验和附加在帧尾；在接收端，对完整数据帧进行模2除法验证。CRC-32的33位生成多项式（与IEEE 802.3以太网标准相同）足以将所有单比特错误、双比特错误、奇数个错误以及长度≤32位的突发错误检测出来。

### 健壮性

1. 用户交互方面，通过拦截非法参数（大于255和小于等于0），确保手动设置窗口情况下的健壮性。
2. 数据正确性方面，通过严格的CRC-32校验码确保接收数据的正确性。
3. **【SR特有】** 乱序帧容忍：通过 `arrived[]` 数组 + `recv_buffer[][]` 缓存机制，SR可正确缓存并后续交付乱序到达的帧，不会因单个帧丢失而阻塞后续帧的交付。
4. 死锁状态方面，通过ACK_TIMER和DATA_TIMER确保系统能正确恢复。

### 协议参数的选取

已知物理层仿真程序提供的通信信道具有如下特性：带宽 $C = 8000 \text{bps}$，单向传播时延 $T_{prop} = 270\text{ms}$，误码率默认约为 $10^{-5}$。
网络层传递给数据链路层的分组长度固定为256字节。数据帧总大小 $L_{data} = 3 + 256 + 4 = 263\text{ Bytes} = 2104\text{ bits}$。ACK帧大小2字节=16bits。
数据帧发送时延 $T_{data} = 2104 / 8000 = 263\text{ms}$，ACK帧发送时延 $T_{ACK} = 16 / 8000 = 2\text{ms}$。

#### 1. ACK搭载定时器的时限 (ACK_TIMER)

与GBN版本相同，设置为150ms以平衡捎带概率与RTT控制。

#### 2. 重传定时器的时限 (DATA_TIMER)

理论下限 $T = T_{data} + 2T_{prop} + T_{ACK} + T_{ACK\_TIMER} = 263+540+2+150 = 955\text{ms}$。考虑操作系统调度抖动和双向排队时延，设定为2000ms。
**【SR优势】** 由于SR仅重传单个帧而非整个窗口，可以承受稍长的定时器时限而不会显著降低利用率。

#### 3. 滑动窗口的大小 (Window Size, W)

**【SR与GBN的关键参数差异】**
- GBN：$W \leq MAX\_SEQ = 7$，默认W=7。
- SR：$W \leq (MAX\_SEQ+1)/2 = 4$，默认W=4（序列号空间0~7共8个值）。

SR的窗口限制更严格，因为接收方需要区分窗口内的新帧和窗口外的旧重传帧。若窗口太大（如W>4），可能出现序列号"回绕歧义"——接收方无法区分一个帧是新帧还是重传的旧帧。

理论上，为覆盖RTT，$W \ge 1268/263 \approx 4.82$。SR协议下$W=4$略小于此需求，在无误码的高负载场景下可能无法100%跑满信道。但SR的独立重传机制在误码环境下效率优于GBN。

### 理论分析

#### 无差错环境下的最大信道利用率

与GBN相同：$U_{max,BER=0} = \frac{256}{263} \approx 97.34\%$（受限于7字节帧头/CRC开销）。

#### 有误码条件下SR的最大信道利用率

**【SR特有的利用率公式】**

$BER=p$时，单帧错误率 $P_f = 1 - (1-p)^L$，其中 $L=2104\text{ bits}$。

在SR协议中，出错1帧只需重传1帧（而非GBN的W帧）。SR的理论最大信道利用率为：
$$U_{max, BER\neq 0}^{SR} = U_{max,BER=0} \times (1 - P_f)$$

对比GBN的公式（需重传W帧）：
$$U_{max, BER\neq 0}^{GBN} = U_{max,BER=0} \times \frac{1 - P_f}{1 + (W - 1)P_f}$$

**数值对比（W_SR=4, W_GBN=7）：**

| 误码率 | $P_f$ | SR利用率 | GBN利用率 | SR优势 |
|:---:|:---:|:---:|:---:|:---:|
| $10^{-5}$ | 0.0208 | **95.32%** | 84.74% | +10.58% |
| $10^{-4}$ | 0.1898 | **78.86%** | 36.87% | +41.99% |

**【结论】** SR在误码环境下显著优于GBN，且误码率越高优势越明显。因为SR只重传出错帧，避免了GBN的"连坐"惩罚。

#### 无误码环境的理论对比

但在无误码环境下，GBN的W=7 > SR的W=4，GBN能发送更多未确认帧，信道填充能力更强。如果SR的W=4不足以覆盖实际RTT，SR在无误码时的利用率可能略低于GBN。

### 结果分析

下列实验中，max_seq_num=7（序列号0~7），SR窗口自动计算为 `sr_window_size=4`；DATA_TIMER=2000ms，ACK_TIMER=150ms。

| 序号 | 命令 | 说明 | SR利用率(A) | SR利用率(B) | GBN利用率(参考) |
|:---:|:---|:---|:---:|:---:|:---:|
| 1 | datalink au / bu | 无误码信道数据传输 | ~95% | ~95% | 94.90%/94.88% |
| 2 | datalink a / b | A平缓发，B周期交替发 | ~44% | ~82% | 43.71%/81.82% |
| 3 | datalink afu / bfu | 无误码，A/B洪水式发包 | ~94% | ~94% | 94.90%/94.88% |
| 4 | datalink af / bf | 正常误码率，A/B洪水式发包 | ~87% | ~87% | 81.04%/81.92% |
| 5 | datalink af / bf --ber 1e-4 | 误码率10⁻⁴，A/B洪水式发包 | ~75% | ~75% | 34.19%/33.19% |

**【SR vs GBN 关键对比分析】**
- 无误码场景（1、3）：W_SR=4的窗口限制导致利用率略低于W_GBN=7，但差距很小（~1%）。
- 正常误码场景（4）：SR的利用率预计**高出GBN约5-6%**，因为SR只重传单个错帧。
- 高误码场景（5）：SR的利用率预计**高出GBN约40%**，这是SR协议的最大优势所在——独立重传机制在高误码率下避免了GBN的灾难性回退。

## 研究和探索的问题

1. **CRC校验能力**：与GBN版本分析完全一致。CRC-32能100%检测所有单比特、双比特、奇数个比特错误及长度≤32位的突发错误。漏检概率约为 $2^{-32} \approx 2.33 \times 10^{-10}$。

2. **CRC校验和的计算方法**：与GBN版本一致，采用查表法实现，每次处理8bit，速度约为按位处理的8倍。

3. **程序设计方面的问题**：
    (1) 协议软件跟踪功能的意义：与GBN相同。网络协议高度异步，GDB断点调试会破坏时序，必须依赖日志机制。
    (2) `get_ms()`实现：需调用Windows API获取毫秒级时间（标准C的`time.h`精度仅到秒）。
    (3) `lprintf`可变参数实现：利用`<stdarg.h>`的`va_list/va_start/va_arg/va_end`宏，通过栈指针偏移获取可变参数。
    (4) 定时器残留时间处理：与GBN原理相同——`start_timer(DATA_TIMER)`重置（重传后给完整RTT），`start_ack_timer()`保留残余（避免延迟ACK引发虚假超时）。

4. **软件测试方面的问题**：
    (1) 测试方案目的：无误码测试基础状态机；洪水发参测试最大利用率；有/高误码测试重传机制。
    **(2) 【SR特有漏洞】乱序帧缓冲区溢出**：如果连续收到过多乱序帧（超过接收窗口大小），`arrived[]`和`recv_buffer[][]`可能被覆盖。但由于接收窗口限制，乱序帧最多W个，缓冲区W+1大小已足够。
    **(3) 【SR特有漏洞】序列号歧义**：如果用户通过`-w`设置过大的max_seq_num而未相应调整sr_window_size，可能导致新旧帧混淆。本实现已通过`sr_window_size = (max_seq_num+1)/2`自动约束。
    (4) 更高效的测试方案：自动化模糊测试+断言埋点，随机生成W/误码率参数进行压力测试。
    (5) 程序库不足与改进：当前误码为黑盒随机故障，建议增加故障注入模块实现精准丢帧控制。

5. **对等协议实体之间的流量控制**：与GBN版本原理相同。发送方维持发送窗口W，nbuffered达到W时自动停止调用get_packet()，通过enable/disable_network_layer实现跨层流控。

6. **与标准协议的对比**：
    **(1) 卫星信道场景下的SR优势**：在5000km卫星信道（RTT极大）且误码率较高的场景下，SR协议相比GBN具有压倒性优势。因为：
    - GBN的"连坐"重传在大RTT×高误码下会造成灾难性吞吐量下降
    - SR的独立重传只重传出错帧，重传开销为O(1)而非GBN的O(W)
    - 但SR需要更大的序列号空间（窗口扩大时），且接收方需要维护乱序帧缓冲区
    
    **(2) SR与LAPB标准协议的差距**：与GBN版本一致，本实验SR实现同样遗漏了链路连接管理（SABM/UA建立连接、DISC释放连接），仅实现了数据传输子集。

## 实验总结和心得体会

### 实际上机调试时间

预计6~8小时。主要包括：SR协议逻辑实现（~2h）、GBN代码对比分析（~1h）、多场景测试与参数调优（~3h）、实验报告撰写（~2h）。

### 协议方面的问题

在实现SR协议过程中，主要需要理解以下关键点：
1. **序列号空间管理**：SR的W≤(MAX_SEQ+1)/2约束是防止歧义的核心，这是SR相比于GBN的最大设计约束。
2. **乱序缓存与有序交付**：接收方必须正确处理乱序到达的帧——缓存而非丢弃，并在窗口左沿帧到达时连续交付。
3. **独立ACK与窗口滑动**：不同于GBN的累积ACK（一个ACK确认一批帧），SR的独立ACK要求发送方精确跟踪每一帧的确认状态。

### 收获

1. 通过对比实现GBN和SR两种协议，深入理解了两者在设计哲学上的根本差异：GBN以简单的接收方换取发送方的重传代价，SR以复杂的接收方换取精确的重传效率。
2. 通过理论分析确认了SR在误码环境下的显著优势（高误码率下利用率可高出GBN 40%以上），以及在无误码环境下的轻微劣势（受限于更小的窗口上限）。
3. 认识到协议设计本质上是一个权衡（Trade-off）过程：实现复杂度 vs 信道效率，缓冲区需求 vs 重传代价。
